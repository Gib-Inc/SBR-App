Complete the QuickBooks Online integration so it is fully usable in production for:
- importing historical demand for forecasting,
- posting completed refunds back to QuickBooks,
- giving a clear Insights view in the AI Agent.

Quick design principles:
- QuickBooks is a FINANCIAL + HISTORY source only.
- Do NOT use QuickBooks as inventory master; Hildale/Pivot quantities stay in our own DB + Extensiv.
- Integration must be one-click visible, testable, and auditable via the Logs tab.

2) DATA MODEL EXTENSIONS
==================================================
2.1 Product mapping to QuickBooks
Extend Product model:

- quickbooksItemId       // string | null, QB itemRef.id
- quickbooksItemName     // optional display name
- quickbooksItemSku      // optional copy of QB item SKU (for debugging)
- upc                    // GS1 UPC already used elsewhere (important for auto-matching)

2.2 Demand history storage
Create table QuickBooksDemandHistory:

- id
- productId (FK → Product)
- quickbooksItemId
- year (int)
- month (int 1–12)
- qtySold (number)        // positive units sold
- qtyReturned (number)    // positive units returned
- netQty (number)         // qtySold - qtyReturned
- revenue (number)        // net revenue for that month
- lastSyncedAt (timestamp)

Index this table on (productId, year, month).

2.3 Returns linkage
Extend Return model (whatever we use for refunds):

- quickbooksRefundId    // QB Credit Memo / Refund Receipt id | null
- quickbooksRefundType  // "CREDIT_MEMO" | "REFUND_RECEIPT" | null

4) DEMAND HISTORY SYNC (QB → APP)
==================================================
4.1 Sync trigger
Implement:
- Manual button: “Sync Demand Now” on the QuickBooks card.
- Background job (e.g. nightly) that runs the same logic if QuickBooks is connected.

Config:
- History window in months: read from AI Agent Rules (see section 5), default 24 months.
- Only process products where product.quickbooksItemId is set.

4.2 Data pulled from QuickBooks
For each mapped QuickBooks item in the selected history window:

- Items endpoint (once) to confirm mapping.
- Transactions:
  • Sales Receipts
  • Invoices
  • Credit Memos
  • Refund Receipts

For each transaction line:
- Identify itemRef.id == product.quickbooksItemId.
- Determine sign:
  • Sales Receipt / Invoice → positive quantity (sale) and positive line amount.
  • Credit Memo / Refund Receipt → negative quantity (return) and negative amount.

Aggregate by (productId, year, month):
- qtySold      = sum of positive quantities.
- qtyReturned  = absolute sum of negative quantities.
- netQty       = qtySold - qtyReturned.
- revenue      = sum of line amounts (sales minus returns).

Upsert into QuickBooksDemandHistory:
- If (productId, year, month) exists → update fields and lastSyncedAt.
- Else → insert new row.

4.3 Logs and status
- After each sync, log a summary entry into the Logs system:
  • total products processed
  • time window
  • number of error rows
- Update “Last demand sync” timestamp on the QuickBooks card.

5) AI AGENT RULES – USING QB HISTORY
==================================================
In AI Agent → Rules tab, add a “QuickBooks Demand History” section:

Components:
- Toggle: “Use QuickBooks demand history for forecasting”
  • When OFF: existing internal forecasting logic only (no QB influence).
  • When ON: forecasting engine should consult QuickBooksDemandHistory wherever available.
- Numeric input: “History window (months)” (e.g. 12–36, default 24).
- Tooltip: explain that this uses historical monthly sales and returns from QuickBooks to adjust seasonality and growth.

Forecasting adjustments (server-side):
- For each product:
  • Read its QuickBooksDemandHistory rows within the window.
  • Compute:
      – monthly average demand,
      – seasonality factor for current month vs average,
      – year-over-year growth where previous-year month exists.
  • Provide a function such as:
      getQuickBooksForecast(productId) → {
        dailyDemand,
        lastYearMonthQty,
        thisYearMonthQty,
        yoyGrowthPercent
      }
- Integrate this into the existing forecast / reorder logic:
  • If QB history exists AND toggle is ON: base forecast should use QB-derived dailyDemand (possibly blended with internal data).
  • If no QB data or toggle OFF: keep current forecast behavior.

7) RETURNS → QUICKBOOKS (APP → QB)
==================================================
7.1 Trigger
When a Return in the app reaches a “Refunded” / “Completed” terminal status:

Preconditions:
- The return is linked to an original sale that is represented in QuickBooks (or at least we know the customer info + items + refund amounts).
- Products in the return are mapped to QuickBooks items (product.quickbooksItemId present).

7.2 Creating financial docs in QuickBooks
For each completed return:

- Decide document type (configurable but default to Credit Memo):
  • If we can link to original invoice → create Credit Memo.
  • Otherwise, use Refund Receipt.

- For each returned line:
  • itemRef.id = product.quickbooksItemId
  • qty = returned quantity (positive number)
  • amount = refund amount (line-level)

- Create the document via QuickBooks API:
  • Associate with the correct customer if possible.

- On success:
  • Save quickbooksRefundId and quickbooksRefundType on the Return.
  • Log success in Logs (type “QuickBooks Refund”, includes return id and QB doc id).

- On failure:
  • Do NOT block internal return completion.
  • Log an error entry with details.
  • Optionally, flag the return row with a small “QB error” indicator the Logs tab can surface later.

7.3 No inventory push
- Do NOT update QuickBooks inventory quantities.
- Our Hildale/Pivot/Extensiv sync remains the single source of inventory truth; QuickBooks is used only for:
  • revenue & COGS correctness,
  • historical demand for forecasting.
