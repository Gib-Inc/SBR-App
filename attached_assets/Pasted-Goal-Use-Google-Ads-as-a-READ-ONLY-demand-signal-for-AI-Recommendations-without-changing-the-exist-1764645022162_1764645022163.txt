2) DATA MODEL – SOURCE DECISION STRUCTURE
==================================================
Extend the AI Recommendation model so each row can carry per-source decisions and explanations.

1. AI Recommendation detail payload:
   - Add a JSON field on the AI recommendation entity, e.g. `sourceDecisionsJson`.
   - Shape (TypeScript-style):

   interface SourceDecision {
     source: "GOOGLE_ADS" | "META_ADS" | "SHOPIFY" | "EXTENSIV" | "QUICKBOOKS";
     status: "ORDER" | "DONT_ORDER" | "NEUTRAL" | "NO_DATA"; // "NO_DATA" means no signal
     rationale: string;                                      // short human-readable reason
     metrics: Record<string, string | number>;               // key metrics used for this source
   }

   interface RecommendationDetail {
     finalDecision: "ORDER" | "DONT_ORDER" | "MONITOR";      // overall call
     finalRationale: string;                                 // 1–3 sentence explanation
     sources: SourceDecision[];
   }

   - Store this as JSON on each AI Recommendation row.

2. Keep existing AI Recommendation fields (SKU, Product, Type, Risk, Days Left, Avail, Gap%, On PO, etc.) intact; they should continue to be first-class columns.

3) GOOGLE ADS – DATA PULL & SOURCE DECISION
==================================================
Implement a read-only Google Ads sync and per-source decision calculation.

1. Data pull (re-use “Sync” from Google Ads card):
   - On sync, pull the last N days (configurable, default 14) of performance per product using Google Ads API:
     • Dimensions: date, campaign/ad group/asset group, product item ID (if available).
     • Metrics: impressions, clicks, cost, conversions, conversion value.
   - Map product item ID → internal product (house SKU) using:
     • existing mapping (SKU/UPC) or
     • any existing Google-item-ID field on Product.
   - Aggregate per product per day, and then per product over the last 14 days:
     • avgConvPerDay
     • avgRevenuePerDay
     • avgSpendPerDay (optional)
   - Store this in a dedicated performance table (if not already created) or cache layer, but keep it READ-ONLY.

2. Google Ads “Order / Don’t order” logic:
   - For each product with Google Ads data and an active AI Recommendation row:
     - Compute:
       • avgConvPerDay = average conversions over the last 7–14 days.
       • projectedOrdersHorizon = avgConvPerDay * daysLeftHorizon,
         where daysLeftHorizon comes from the recommendation’s forecast window.
       • available = current available-for-sale units (same as used in AI Recommendations).
     - Define a simple rule (you can adjust thresholds if more sensible):
       • If projectedOrdersHorizon > available * 0.7 → Google Ads status = "ORDER"
       • Else if projectedOrdersHorizon < available * 0.3 → status = "DONT_ORDER"
       • Else → status = "NEUTRAL"
     - If there is no mapped Google Ads data for that product → status = "NO_DATA".

   - Build a SourceDecision for "GOOGLE_ADS" with:
     • status (Order / Don’t order / Neutral / No data),
     • rationale string, e.g.:
       - "High ad-driven demand: ~X orders expected vs Y units available."
       - "Ad-driven demand appears low relative to current stock."
       - "No recent Google Ads data available for this SKU."
     • metrics object including:
       - avgConvPerDay
       - projectedOrdersHorizon
       - available
       - daysLeftHorizon
       - time window used (e.g., "last_14_days").

3. DO NOT WRITE BACK to Google Ads:
   - No changes to bids, budgets, or campaign settings.
   - This source is strictly used to inform recommendations.
