==================================================
C) RETURNS SERVICE – CORE LOGIC
==================================================
Create or extend a ReturnsService with methods:

1) requestReturn(payload)
   Input (from GHL agent):
   - customerId / customerName / customerEmail
   - channel
   - orderId
   - items: [{ sku, productName, quantity }]
   - reasonCode
   - reasonText
   - desiredResolution ("REFUND" or "REPLACEMENT" etc.) if provided; default REFUND for V1.

   Behavior:
   - Validate inputs; ensure orderId exists in Sales Orders.
   - Create RETURNS row with:
     - status = REQUESTED
     - resolution set as passed or default REFUND.
     - timestamps: requestedAt = now.
   - Create RETURN_ITEMS rows.
   - Log RETURN_EVENT type="GHL_REQUEST".
   - For V1, auto-approve most returns:
     - Set status = APPROVED
     - approvedAt = now.
   - Call createReturnLabel(returnId) immediately if resolution requires physical return (REFUND/REPLACEMENT).
   - Update linked Sales Order:
     - If no existing returns: set returnStatus = "IN_PROGRESS".
     - Increase totalReturnQty by sum of return_items.quantity for this return.

   ReturnsService.requestReturn should return:
   - { returnId, rmaNumber, status, resolution, orderId }

2) createReturnLabel(returnId)
   Behavior:
   - Load return + items + customer shipping address (from order data).
   - If status is not APPROVED, throw.
   - Build Shippo Shipment and purchase a RETURN label:
     - address_from = warehouse address.
     - address_to = customer’s shipping address.
     - Mark shipment as return (use Shippo’s return flag/field).
   - On success:
     - Store shippoShipmentId, shippoTransactionId, carrier, trackingNumber, labelUrl, labelCost, labelCurrency.
     - Set status = LABEL_CREATED.
     - labelCreatedAt = now.
     - Log RETURN_EVENT type="LABEL_CREATED" with payload.
   - Return:
     - rmaNumber, labelUrl, trackingNumber, carrier.

3) updateStatusFromTracking(returnId, newStatus, payload)
   - Map carrier/Shippo statuses to our internal:
     - PRE_TRANSIT / TRANSIT → IN_TRANSIT (set inTransitAt).
     - DELIVERED → RETURNED (set receivedAt).
   - Update returns.status, relevant timestamps.
   - Log RETURN_EVENT type="SHIPPO_WEBHOOK".

4) markRefundIssuePending(returnId)
   - Called when the system decides a refund is needed (for REFUND resolution) and label is created or package is returned.
   - Set status = REFUND_ISSUE_PENDING.
   - Log RETURN_EVENT type="STATUS_CHANGE".
   - Call syncReturnRefundTaskToGhl(returnId) (section D) to create/update the “Issue refund” Opportunity.

5) markRefundCompleted(returnId)
   - Called by an internal UI action AFTER a team member actually moves money in Amazon/Shopify.
   - Set status = REFUNDED.
   - refundedAt = now.
   - Update linked Sales Order:
     - If all return quantity for that order is refunded, returnStatus = "REFUNDED".
     - If partial, returnStatus = "PARTIAL_REFUNDED".
   - Log RETURN_EVENT type="STATUS_CHANGE".
   - Call syncReturnRefundTaskToGhl(returnId) to move Opportunity stage to REFUNDED.
