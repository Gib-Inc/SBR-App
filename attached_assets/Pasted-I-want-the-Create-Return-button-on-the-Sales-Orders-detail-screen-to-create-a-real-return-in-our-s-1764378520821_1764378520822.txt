I want the “Create Return” button on the Sales Orders detail screen to create a real return in our system and generate a valid Shippo return label, then track that return on the Returns page.

Please inspect the existing SalesOrder, Returns, and Shippo integration code first, reuse anything that already exists (models, endpoints, UI patterns), and only add what is missing.

==================================================
A) DATA MODEL – RETURN ORDER + RETURN LINES
==================================================
1) Check the current schema for returns.
   - If we already have ReturnOrder/ReturnLine (or equivalent) models, extend them as needed.
   - If not, create:

   ReturnOrder:
   - id (PK)
   - salesOrderId (FK to SalesOrder)
   - channel (e.g. "SHOPIFY", "AMAZON")
   - customerName
   - email
   - phone
   - shipFromAddress (JSON or fields for name, street, city, state, zip, country)
   - shipToAddress (warehouse address)
   - status enum: "CREATED" | "IN_TRANSIT" | "RECEIVED" | "CLOSED"
   - reason (text, nullable)
   - shippoShipmentId (string, nullable)
   - shippoTransactionId (string, nullable)
   - labelUrl (string, nullable)
   - trackingNumber (string, nullable)
   - carrier (string, nullable)
   - serviceLevel (string, nullable)
   - createdAt, updatedAt

   ReturnLine:
   - id (PK)
   - returnOrderId (FK)
   - salesOrderLineId (FK to SalesOrderLine)
   - sku
   - productName
   - qtyOrdered
   - qtyReturning
   - unitPrice
   - reason (nullable)

2) Ensure SalesOrderLine has:
   - returnedQty (number, default 0)

==================================================
B) BACKEND – CREATE RETURN FROM SALES ORDER
==================================================
Create a new endpoint, reusing existing patterns for auth and validation:

- POST /api/returns/from-sales-order

Input:
- salesOrderId: string
- lines: Array<{ salesOrderLineId: string; qtyToReturn: number; reason?: string }>
- overallReason?: string
- serviceLevelToken?: string (optional for choosing Shippo service level)

Server flow:
1) Load SalesOrder and its lines.
2) For each input line:
   - Validate that the line belongs to the order.
   - Validate qtyToReturn > 0.
   - Validate qtyToReturn <= (fulfilledQty - returnedQty) for that line.
   - If any check fails, return a 400 with a clear message.
3) Build a new ReturnOrder object:
   - salesOrderId, channel, customerName, email, phone.
   - shipFromAddress = customer shipping address from the order.
   - shipToAddress = warehouse address from config/env (WAREHOUSE_* env vars if available; if not, create them).
   - status = "CREATED"
   - reason = overallReason.
4) Build ReturnLine records for each line with:
   - salesOrderLineId, sku, productName, qtyOrdered, qtyReturning, unitPrice, reason.
5) Call a Shippo integration helper to create a return shipment + transaction:
   - Inputs:
     - from: shipFromAddress (customer)
     - to: shipToAddress (warehouse)
     - parcels: for V1, you can use a sensible default parcel definition or derive from line items if that already exists.
     - serviceLevelToken: use the provided token or a default from env.
   - On success:
     - Save shippoShipmentId, shippoTransactionId, labelUrl, trackingNumber, carrier, serviceLevel on ReturnOrder.
   - On error:
     - Do NOT commit the ReturnOrder/ReturnLine changes.
     - Return an error to the client.
     - Log a system log event type "SHIPPO_ERROR" with details.

6) On success:
   - Increment SalesOrderLine.returnedQty for each line by the qtyToReturn.
   - Optionally update SalesOrder status to something like "RETURN_IN_PROGRESS" if that fits our status model.
   - Return the ReturnOrder plus its lines and labelUrl to the client.
