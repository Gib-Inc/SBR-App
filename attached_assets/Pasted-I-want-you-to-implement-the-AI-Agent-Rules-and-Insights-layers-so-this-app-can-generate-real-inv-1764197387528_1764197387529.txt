Goals:
• Centralize inventory logic in a single decision engine that computes:
– Reorder recommendation per finished product and key raw materials.
– Recommended order quantity.
– Risk level (LOW / MEDIUM / HIGH).
– Key drivers (sales velocity, lead time, backorders, returns rate, supplier reliability, ads demand).

Decision engine module (backend only)
• Create a dedicated service/module (e.g. InventoryDecisionEngine) that:
– Reads data from existing tables:
· Products & BOM (finished vs components, MOQ, lead time, unit cost).
· Stock Inventory (current on-hand, safety stock if present).
· PurchaseOrders (open POs, expected dates, issue/dispute flags).
· SalesOrders (fulfilled units, daily/weekly sales velocity, backorders).
· Returns (return rate per SKU, refund vs replacement ratios).
· Extensiv snapshots (3PL stock) for any SKUs mapped to external warehouse.
– Computes per SKU:
· dailySalesVelocity (smoothed over last N days).
· projectedDaysUntilStockout = (onHand + inboundPO) / dailySalesVelocity (guard for zero).
· effectiveLeadTime = supplierLeadTime + buffer days if disputes/late POs are common.
· recommendedReorderPoint = dailySalesVelocity * effectiveLeadTime + safetyStock.
· recommendedReorderQty = max(recommendedReorderPoint – (onHand + inboundPO), 0) rounded to MOQ.
· returnRate = returnsUnits / shippedUnits, used to DOWN-WEIGHT reorderQty when returns are high.
· riskLevel:

HIGH if projectedDaysUntilStockout < leadTime or backorders > 0.

MEDIUM if close to reorder point or supplier reliability is poor.

LOW otherwise.
– Returns a structured object per SKU with:
· sku, productName.
· recommendedAction: "ORDER", "WATCH", or "NO_ACTION".
· recommendedQty.
· riskLevel.
· metrics snapshot (velocity, daysUntilStockout, onHand, inboundPO, returnRate, supplierScore).
· human-readable explanation string, e.g.:
“Order 300 units: on-hand 120, inbound 0, selling 10/day, 4-day lead time, no active disputes.”
