Please inspect the existing implementation and then fix it so:

- The PO list table “Status” column and
- The PO detail header “Status” badge

both use the same derived status value from the same server-side logic.

==================================================
A) FIND CURRENT STATUS SOURCES
==================================================
1) Inspect:
   - The API that returns the Purchase Orders list.
   - The API that returns a single Purchase Order for the detail view.
   - The React components for:
     - Purchase Orders table/list.
     - Purchase Order detail modal.

2) Identify:
   - How the list “Status” label is currently computed (enum value, helper, etc.).

==================================================
B) SERVER-SIDE STATUS DERIVATION (SINGLE SOURCE OF TRUTH)
==================================================
If not already done, create or reuse a server-side helper:

- `derivePoStatus(po: PurchaseOrder, lineItems: PurchaseOrderLine[]): PurchaseOrderStatus`

that applies the lifecycle we already agreed on:

Priority order:
- If DRAFT and no email sent → "Draft"
- Else if some but not all qtyReceived < qtyOrdered → "Partial"
- Else if all qtyReceived == qtyOrdered and qtyOrdered > 0 → "Received"
- Else if accepted (supplier or internal) and no receipts yet → "Accepted"
- Else if at least one successful email send and no acceptance/receipts yet → "Sent"
- Else → fallback to "Draft" or existing default.

Use:
- totalQtyOrdered = sum(line.qtyOrdered)
- totalQtyReceived = sum(line.qtyReceived)
- accepted flag(s) from existing acknowledgement fields or internal-accepted flag.
- email sent indicator from existing email status.

Apply this helper:
- When preparing data for the PO list API.
- When preparing data for the PO detail API.

Expose the result as a single field, e.g. `derivedStatus` or `status`, consistently in both APIs.
