Update ORM/typescript models and API types accordingly.

==================================================
B) PO STATE MACHINE + API
==================================================
Implement a state machine for purchase_orders.status:

- Valid transitions:
  - DRAFT → APPROVED
  - APPROVED → SENT
  - SENT → PARTIALLY_RECEIVED
  - PARTIALLY_RECEIVED → RECEIVED
  - RECEIVED → CLOSED
  - Any open state → CANCELLED (with reason)

API endpoints (REST or equivalent):

1) POST /api/purchase-orders
   - Create a new PO in DRAFT with header + line items.
   - Autogenerate poNumber (sequential) using a central helper.

2) GET /api/purchase-orders
   - List POs with filters: status, supplierId, date range.

3) GET /api/purchase-orders/:id
   - Return header, lines, derived totals, and receipts.

4) PATCH /api/purchase-orders/:id
   - Update header fields (buyer/supplier info, dates, terms).
   - Change status using the allowed transitions only.

5) POST /api/purchase-orders/:id/lines
   - Add or update line items (sku, orderedQty, unitCost, etc.).

6) POST /api/purchase-orders/:id/receipts
   - Body: dateReceived, warehouseLocation, and an array of { sku, receivedQty }.
   - Logic:
     - For each line:
       - Increase line.receivedQty by sum of receipts for that sku.
       - Cap at orderedQty.
     - Update purchase order status:
       - If some but not all orderedQty are received: PARTIALLY_RECEIVED.
       - If all orderedQty are received: RECEIVED.
     - Update inventory tables to add receivedQty to on-hand stock for each sku.

==================================================

     - Include in GHL custom fields:
       - poNumber
       - poTotal (total)
       - poStatus (mirrors our status)

2) Add an endpoint to receive approval updates from GHL:

       - Change status from DRAFT → APPROVED.

3) Provide a URL for each PO detail:
   - e.g. /purchase-orders/:id
   - Use this as the deep link from GHL so users can open the full PO view in this app.
