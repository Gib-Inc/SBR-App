====================================================
2. BACKEND: ADD MASKED PREVIEW FIELDS (NEVER RETURN FULL SECRET)
====================================================

Objective: The GET endpoint for LLM settings must return only masked previews and flags indicating whether a secret exists. The full secrets remain on the server only.

2.1 Add a masking helper on the server

- Create or update a small helper function, in a shared server utility file (for example server/utils/secrets.ts).
- The helper is called maskSecret and behaves as follows:
  - Input: a string secret (or null/undefined).
  - If the value is null or empty, return null.
  - Otherwise, trim whitespace.
  - If the trimmed string length is 6 characters or less, return a string composed of the first character, then three asterisks, then the last character (for example “a***z”).
  - If the trimmed string is longer than 6 characters, return a string composed of:
    - the first 3 characters,
    - then three asterisks,
    - then the last 3 characters.
    - Example: “sk-longapikey123” becomes “sk-***123”.
  - This function must never log or otherwise expose the original secret.

2.2 Update the LLM config GET handler

- In the LLM settings GET route (for example /api/settings/llm-config):
  - Load the full configuration from storage as you currently do, including the real openAiApiKey and openAiWebhookSigningSecret values.
  - When building the JSON response for the client:
    - Do NOT include the real openAiApiKey.
    - Do NOT include the real openAiWebhookSigningSecret.
  - Instead, include the following additional fields in the JSON output:
    - hasApiKey: a boolean that is true if a real API key is present, false otherwise.
    - apiKeyMasked: the result of calling maskSecret on the real API key (or null).
    - hasWebhookSigningSecret: a boolean that is true if a webhook signing secret is present.
    - webhookSigningSecretMasked: the result of calling maskSecret on the real webhook signing secret (or null).
  - All other non-secret config values (provider, default model, temperature, max tokens, etc.) should be returned as before.
  - Confirm that nowhere in this handler is the full key or secret added to the JSON or logs.

2.3 Update the LLM config UPDATE handler

- In the POST or PUT route that saves the LLM configuration:
  - Read the incoming body which contains provider, defaultModel, temperature, maxTokens and optionally apiKey and webhookSigningSecret.
  - Before saving, load the existing configuration from storage so you know the currently stored secrets.
  - For the API key:
    - If the request body does not include an apiKey field at all, or if the apiKey is an empty string after trimming, then keep the existing stored API key unchanged.
    - If the request includes a non-empty apiKey string, trim it and overwrite the stored openAiApiKey with this new value.
  - For the webhook signing secret:
    - Use the same rule: only overwrite if a non-empty string is provided; otherwise keep the existing stored secret.
  - Save the updated configuration with:
    - provider, defaultModel, temperature, maxTokens updated from the request,
    - openAiApiKey set to either the previous or the new value as described,
    - openAiWebhookSigningSecret set similarly.
  - Ensure there are no console logs or logger statements that print the full secret values. If any exist, remove them or change them to use the masked form only.

====================================================

Objective: The LLM Configuration tab should show the masked preview text underneath each secret input, while the input fields themselves stay empty unless the user types a new value. The masked text must not be bound to the input value.

  - hasWebhookSigningSecret (boolean)
  - webhookSigningSecretMasked (string or null)
- In the React component for the LLM Configuration tab:
  - Keep a piece of state for the entire settings object returned from the server.
  - Additionally, keep separate local pieces of state for the editable secret inputs, for example:
    - localApiKey (string)
    - localWebhookSecret (string)
  - Initialize localApiKey and localWebhookSecret as empty strings when the component mounts.

3.2 Render the “OpenAI API Key” field

- The text input for the API key should:
  - Use type=password.
  - Have its value bound to localApiKey only.
  - Update localApiKey when the user types.
  - Have a placeholder that depends on whether a key already exists:

    - If hasApiKey is false, use a placeholder like: “Enter API key from platform.openai.com”.
- Under the input, render a small read-only line when there is a saved key:
  - If hasApiKey is true and apiKeyMasked is not null, show something like:
    - “Saved key: sk-***b0A” using the apiKeyMasked string from the API.
  - This line is purely informational; it is not an input and cannot be edited.

3.3 Render the “OpenAI Webhook (signing secret)” field

- Use the same pattern as the API key:
  - The input’s value is bound only to localWebhookSecret.
  - The placeholder depends on whether a secret exists:
    - If hasWebhookSigningSecret is true, placeholder like: “Enter new signing secret to replace existing”.
    - If false, placeholder like: “Enter webhook signing secret”.
  - Under the input, if hasWebhookSigningSecret is true and webhookSigningSecretMasked is present, render:
    - “Saved signing secret: whs***XYZ” using the masked value from the API.
  - Again, this masked value is for display only, not the input value.

3.4 Save behavior for the LLM configuration form

- When the user clicks “Save Configuration”:
  - Construct the payload for the LLM settings update as follows:
    - Always include non-secret configuration values that the user can change: provider, defaultModel, temperature, maxTokens.
    - Include apiKey in the payload ONLY if localApiKey, after trimming, is a non-empty string.
    - Include webhookSigningSecret in the payload ONLY if localWebhookSecret, after trimming, is a non-empty string.
  - Send this payload to the LLM settings update API route.
- After a successful save:
  - Clear localApiKey and localWebhookSecret back to empty strings so the inputs are blank again.
  - Re-fetch the LLM settings from the API, update the settings state, and therefore refresh hasApiKey, apiKeyMasked, hasWebhookSigningSecret, and webhookSigningSecretMasked.
  - This ensures that after a change, the user sees the new masked previews.
