SECTION 1 – BUSINESS RULES FROM STICKER BURR ROLLER POLICY
=====================================================================

Match the current policy at https://www.stickerburrroller.com/pages/return-policy:

1) Return window:
   - Customer may return non-defective products within 30 days of delivery.
   - If more than 30 days have passed since delivery, the order is NOT REFUNDABLE and no label should be created.

2) Condition & fees:
   - Unused, clean, sellable condition:
     - Eligible for full product refund, MINUS original shipping paid.
   - Used/dirty/damaged but still returnable:
     - Apply a 20% restocking fee on the product amount.
   - Not returnable (heavily used or damaged beyond policy):
     - No refund.

3) Shipping costs:
   - Outbound shipping from the original order is never refunded.
   - If the customer uses a prepaid return label we provide (via Shippo):
     - Deduct the cost of that label from the refund.
     - Use a MINIMUM fee of $30 (configurable) to stay within policy. If Shippo’s actual label cost is greater than $30, deduct the actual cost.
   - If the customer arranges their own shipping (not V1 focus), there is no label fee deduction.

4) Refund timing:
   - Refunds are only processed AFTER the product is received and inspected at the Hildale warehouse.
   - Policy says refunds should be issued within ~2–5 business days of receipt; in V1 this is a human responsibility. The app should only calculate and recommend the refund amount and create a “Needs Attention” task in GHL.

5) V1 automation boundary:
   - The app MUST NOT issue refunds automatically (no automatic QuickBooks Refund Receipt / Credit Memo in V1).
   - Instead, after warehouse scan and refund calculation, the app must:
     - Compute final refund amount (product subtotal only, minus shipping, label cost, and restocking fee).
     - Create or update a GoHighLevel Opportunity in the “Replit” pipeline, “Needs Attention” stage, instructing a human to issue the refund manually in the correct system (e.g., Amazon, Shopify, etc.).

SECTION 4 – GHL → RETURNS ENDPOINT (ELIGIBILITY + LABEL CREATION)
=====================================================================

Create a single endpoint that GHL or a GHL AI Agent can call when a customer says “I want a refund.”

1) Endpoint: POST /api/returns/from-ghl
   - Authentication: same as other internal APIs (require auth / API token).
   - Request body:
     - ghlContactId
     - externalOrderId (e.g., Shopify/Amazon order id)
     - salesChannel: 'SHOPIFY' | 'AMAZON' | 'DIRECT' | 'OTHER'
     - customerName
     - customerEmail
     - customerPhone
     - shippingAddress (full)
     - items: array of { sku, qtyRequested, unitPrice }
     - reason (string)

2) Return eligibility logic:
   - Find the underlying sales order by externalOrderId + salesChannel (or fail gracefully if not found).
   - Determine deliveryDate:
     - Prefer stored `deliveredAt` on sales order if present.
     - Else fallback to `shippedAt` + typical transit days (configurable, e.g., 5 days).
   - Compute daysSinceDelivery = today - deliveryDate.
   - If daysSinceDelivery > 30:
     - DO NOT create return_requests.
     - Return JSON:
       - eligible: false
       - reasonCode: 'OUTSIDE_RETURN_WINDOW'
       - messageText: clear explanation that the order is past the 30-day return window and cannot be refunded.
     - Stop.

3) If eligible (<= 30 days):
   - Create return_requests row:
     - status = 'REQUESTED'
     - resolution_requested = 'REFUND'
     - labelProvider = 'SHIPPO'
     - source = 'GHL'
     - ghlContactId, externalOrderId, salesChannel, customerName, customerEmail, customerPhone
     - shippingAddress
     - store link to salesOrderId if available.
   - Create return_items rows:
     - For each item: sku, qtyRequested, unitPrice.
   - Emit return_events row of type 'GHL_REQUEST'.
   - For V1, auto-approve straightforward non-defective returns:
     - status = 'APPROVED'
     - return_events: 'STATUS_CHANGE' (REQUESTED → APPROVED).
   - Call helper createReturnLabelForRequest(returnRequestId) (next section).
   - Respond to caller with:
     - eligible: true
     - messageText for GHL agent to send to customer (plain language summary of:
       - 30-day window
       - shipping not refunded
       - possible restocking fee for used items
       - return label cost will be deducted)
     - labelUrl
     - carrier
     - trackingNumber
     - returnRequestId

SECTION 7 – WAREHOUSE SCAN, INVENTORY UPDATE, REFUND CALCULATION
=====================================================================

Warehouse scan is the critical moment when the product is actually received and inspected. This is the only moment when refunds are calculated.

1) Endpoint: POST /api/shippo/scan
   - Request body:
     - scanCode (string; typically trackingNumber barcode)
     - scannedBy (user id / name)
     - per-item condition info:
       - items: [{ returnItemId, condition: 'GOOD_UNUSED' | 'USED_RETURNABLE' | 'NOT_RETURNABLE', qtyReceived }]
   - Steps:
     1) Find shippo_label_logs by scanCode.
        - If not found, return 404.
     2) Load associated return_requests + return_items.
     3) Update shippo_label_logs:
        - status = 'SCANNED_RECEIVED'
        - scannedAt = now()
        - scannedBy.
     4) Update return_requests:
        - status = 'RETURNED'
        - receivedAt = now()
     5) Inventory adjustments (Hildale only):
        - For each items entry:
          - If condition is GOOD_UNUSED or USED_RETURNABLE:
            - Insert inventory_transactions row:
              - itemId, type = 'RECEIVE', location = 'HILDALE', quantity = qtyReceived, notes referencing return + trackingNumber.
            - Increase items.hildaleQty by qtyReceived.
          - If NOT_RETURNABLE:
            - No inventory increment; optionally insert a SCRAP transaction type.
     6) Refund math:
        - Determine productSubtotal:
          - For each return_items row, use unitPrice * qtyReceived.
          - Do NOT include original shipping charges (shipping is not refunded).
        - Outbound shipping amount:
          - If sales order has a separate shippingAmount, include it in logs but not in refund.
        - Restocking fee:
          - For items where condition == GOOD_UNUSED: restocking fee = 0 on those lines.
          - For items where condition == USED_RETURNABLE:
            - Apply 20% of the product line subtotal for those items.
        - Label fee:
          - Use actual labelCost from return_requests.
          - Also load configured RETURN_LABEL_MIN_FEE_USD (default 30).
          - labelFeeAmount = max(labelCost, RETURN_LABEL_MIN_FEE_USD).
        - Compute:
          - baseRefundAmount = productSubtotal (sum of product line subtotals).
          - totalDeduction = restockingFeeAmount + labelFeeAmount.
          - finalRefundAmount = max(baseRefundAmount - totalDeduction, 0).
          - If all items are NOT_RETURNABLE, set finalRefundAmount = 0 and mark as non-refundable.
     7) Persist on return_requests:
        - baseRefundAmount
        - restockingFeeAmount
        - labelFeeAmount
        - finalRefundAmount
        - currency
        - conditionSummary / notes
        - status = 'REFUND_CALCULATED'
     8) Emit return_events:
        - 'WAREHOUSE_SCAN'
        - 'ITEM_RECEIVED' per item
        - 'REFUND_CALCULATED'
        - 'STATUS_CHANGE' (RETURNED → REFUND_CALCULATED)
