2) GHL SYNC LOGIC (PIPELINE / OPPORTUNITIES)
========================================================

Use the existing GHL connection (location ID, pipeline ID, stage IDs) and whatever structure we already have.

Signature idea:
`syncGHL({ mode })` → `{ mode, dealsUpdated, dealsArchived, tasksUpdated }`

A. Common behavior (merge & replace)
- Pull data from GHL:
  - For the configured pipeline / stages (Replit pipeline and PO / refund stages already defined).
  - Fetch relevant fields:
    - Opportunity ID (GHL ID)
    - Pipeline ID, Stage ID
    - Contact info (name, email, phone)
    - Monetary value / amount
    - Custom fields we already use for linking to orders / POs / refunds
- Upsert into our internal tables:
  - For “PO Sent / PO Paid / PO Delivered / Sales Orders / Refunded / Processing Refund”:
    - If an opportunity with that GHL ID already exists, update it.
    - Otherwise create a new record in our GHL-linked tables.
- Never touch non-GHL data.

B. mode = "merge" (Import only)
- Only upsert:
  - Don’t delete or archive any local GHL opportunities that no longer exist upstream.
- Counts:
  - `dealsUpdated` = created + updated.
  - `dealsArchived` = 0.
  - `tasksUpdated` = any internal tasks/flags we update in our app.

C. mode = "replace" (Replace with GHL)
- After importing all current GHL opportunities in scope:
  - Find local GHL-linked opportunities (those with a GHL ID / pipeline = Replit pipeline).
  - Any local opportunity that:
    - Has `channel = GHL` (or whatever flag we use), AND
    - No longer appears in the latest GHL data
    → Archive or soft-delete it (follow existing convention).
- Do NOT delete:
  - Internal records that are not tied to a GHL ID.
  - Any other channels (Shopify, Amazon, manual, etc.).
- Counts:
  - `dealsUpdated` = upserted deals.
  - `dealsArchived` = GHL deals archived/removed.
  - `tasksUpdated` = tasks or flags updated.

D. Logging
- Log each run with:
  - source: "GHL"
  - mode, timestamp
  - counts above
  - any GHL API errors.

3) AMAZON SELLER CENTRAL SYNC LOGIC
========================================================

This is for marketplace orders and inventory that we map into our Sales Orders and BOM.

Signature idea:
`syncAmazon({ mode })` → `{ mode, ordersUpdated, ordersArchived, inventoryUpdated, inventoryMappingsCleared }`

A. Common behavior (merge & replace)
- Using our existing Amazon connection:
  - Fetch Amazon orders:
    - Order ID
    - Status (open, shipped, cancelled, returned, etc.)
    - Customer info
    - Line items: SKU/ASIN, quantity, price
  - Fetch inventory levels for the configured Amazon channels (FBA/FBM as we defined).
- Upsert:
  - Sales Orders table with channel = AMAZON.
    - Map Amazon SKUs/ASINs → our products (using canonical SKU / UPC / mapping wizard).
  - Update Amazon-related inventory fields:
    - e.g., Amazon on-hand, reserved, inbound, etc., if we already track them.
- Do NOT touch non-Amazon channels.

B. mode = "merge" (Import only)
- No deletions:
  - Keep local Amazon orders that don’t show up in this sync.
  - Keep any Amazon inventory mappings that don’t exist upstream.
- Counts:
  - `ordersUpdated`, `ordersArchived = 0`, `inventoryUpdated`, `inventoryMappingsCleared = 0`.

C. mode = "replace" (Replace with Amazon)
- Orders:
  - After pulling all Amazon order IDs in scope:
    - Find local Sales Orders with channel = AMAZON.
    - If an Amazon order in our DB is not present in the Amazon response:
      - Archive / soft-delete / mark as “Archived (no longer in Amazon)” according to existing pattern.
- Inventory:
  - For any Amazon-linked inventory rows that refer to SKUs no longer existing in Amazon:
    - Clear or mark the Amazon-specific link as inactive (do NOT delete the base product).
- Counts:
  - `ordersUpdated`, `ordersArchived`, `inventoryUpdated`, `inventoryMappingsCleared`.

D. Logging
- Source: "Amazon"
- mode, timestamp, counts, error messages.

4) EXTENSIV/PIVOT (3PL) SYNC LOGIC
========================================================

Extensiv is the 3PL that feeds the “Pivot” quantities in BOM. It should NEVER directly affect Hildale quantities; those are only updated via transfers and shipments we define in the app.

Signature idea:
`syncExtensiv({ mode })` → `{ mode, inventoryUpdated, inventoryCleared }`

A. Common behavior (merge & replace)
- Fetch from Extensiv:
  - All SKUs and on-hand quantities for the Pivot warehouse/location.
- Mapping:
  - Map Extensiv SKUs → our canonical products (using the mapping system we already built).
- Upsert:
  - Only the Pivot quantity fields in the BOM Finished Products table.
  - Do NOT touch Hildale Qty; only update Pivot Qty.
  - Do NOT touch Shopify / Amazon / other source fields here.

B. mode = "merge" (Import only)
- Behavior:
  - Only update Pivot Qty for SKUs returned by Extensiv.
  - Leave existing Pivot Qty for SKUs that Extensiv didn’t mention.
- Counts:
  - `inventoryUpdated` = number of Pivot Qty fields updated.
  - `inventoryCleared = 0`.

C. mode = "replace" (Replace with Extensiv)
- Treat Extensiv as the ground truth for Pivot:
  - After syncing all Extensiv SKUs:
    - For any product that has a Pivot Qty in our app but is not present in Extensiv:
      - Set Pivot Qty to 0 and/or mark the Extensiv link as inactive.
- Still NEVER change Hildale Qty here.
- Counts:
  - `inventoryUpdated` = updated Pivot Qty values.
  - `inventoryCleared` = number of Pivot Qty fields that were zeroed or unlinked.

D. Logging
- Source: "Extensiv"
- mode, timestamp, counts, any Extensiv errors.

5) QUICKBOOKS ONLINE SYNC LOGIC
========================================================

QuickBooks is used for financial & demand history, NOT for real-time inventory control. We DO NOT push inventory back to QuickBooks. This sync is read-only from QBO to our “Quickbooks Demand History” table under AI Agent → Insights.

Signature idea:
`syncQuickBooks({ mode })` → `{ mode, recordsUpdated, recordsCleared }`

A. Common behavior (merge & replace)
- Pull from QuickBooks:
  - Sales invoices / sales receipts for products we track:
    - Date, product, quantity, revenue.
  - Credit memos / refunds that affect demand.
- Aggregate or normalize:
  - Create or update entries in our QuickBooks Demand History table(s):
    - SKU / product
    - Period (e.g., month/year)
    - Units sold
    - Units refunded
    - Net demand
    - Revenue, if relevant.
- This table is read-only from the app’s perspective and used by AI in the “Quickbooks Demand History” tab under Insights.

B. mode = "merge" (Import only)
- Behavior:
  - Upsert history rows based on SKUs and time buckets (e.g., year+month).
  - If a period already exists, update totals.
  - Do NOT delete any rows, even if they no longer appear in QuickBooks (safe option).
- Counts:
  - `recordsUpdated` = rows inserted or updated.
  - `recordsCleared = 0`.

C. mode = "replace" (Replace with QuickBooks) — still conservative
- Behavior:
  - Use QBO as the source of truth for the QuickBooks Demand History table ONLY.
  - After importing:
    - Identify rows in the QuickBooks Demand History table that:
      - Are tagged as `source = "QuickBooks"` AND
      - No longer correspond to any QuickBooks transactions in the current sync window.
    - Remove or mark those rows as archived.
  - Never delete or change any core product or inventory tables; only clean up the QuickBooks history table.
- Counts:
  - `recordsUpdated` = rows inserted or updated.
  - `recordsCleared` = history rows removed/archived.

D. Logging
- Source: "QuickBooks"
- mode, timestamp, counts, and any QuickBooks API errors.
