## B-004 — Raw Materials Daily Use Fix

**Severity:** Critical (blocks Reports and reorder signals).

### Root cause

The server logic in `server/routes.ts:21486-21571` is correct on paper: it pulls sales velocity from `storage.getSkuSalesVelocity(90)`, multiplies by BOM quantities, and derives `dailyUsage` per component.

**Verified 2026-04-23:** `storage.getSkuSalesVelocity` at `server/storage.ts:6124` already queries live from `sales_order_lines` JOIN `sales_orders` filtered by `order_date >= cutoff`. No stale-table dependency. This step is a non-issue — the velocity calc is healthy.

Remaining issues:

1. `items.daily_usage` is persisted zero everywhere — nothing is writing it (the column is computed at request time).
2. `items.current_stock` is zero for 43 of 66 items — missing source of truth for raw materials and possibly out-of-sync finished goods.
3. Unknown how many `items` rows are correctly tagged with `type='component'` (the dashboard filter). Needs a quick check before any fix work.

### Fix plan

**Step 1 — Audit `items.type` tagging.** Run `SELECT type, count(*) FROM items GROUP BY type`. Confirm 34 components (matching bom_items distinct materials) carry `type='component'`. If fewer, tag them via a one-time UPDATE against the bom_items.material_sku list. Blocker for everything else in B-004.

**Step 2 — Decide whether `items.daily_usage` is a stored column or derived.**
- **Option A (recommended):** Treat `items.daily_usage` as derived-only. Remove it from writes; leave column for now. Raw Materials dashboard already computes at request time. Downstream (Reports, reorder signals) must also call `storage.getSkuSalesVelocity` rather than reading the column. Lower write amplification, no drift risk.
- **Option B:** Materialize `items.daily_usage` via a scheduled job (hourly via existing `scheduler-service.ts`). Dashboard and Reports can read the column. Simpler consumer code, but a refresh bug will silently poison every downstream reader.

**Recommendation:** Option A. It matches what the dashboard already does. If Reports needs the number in bulk, add a SQL view `v_item_daily_usage` that computes it on read.

**Step 3 — Fix `items.current_stock` population for the 43 of 66 items showing zero.** Two sub-paths:
- Components (raw materials): Sammie enters via the existing Pencil edit in `raw-materials.tsx` (PATCH `/api/items/:id` with `currentStock`). This path works. Deliver a one-page worksheet generated from `items WHERE type='component' AND current_stock IS NULL` for her to fill in. Keep manual entry as the source of truth for raw materials.
- Finished products: `items.current_stock` should come from the Extensiv sync (`extensiv-sync.js`) for warehouse-held inventory plus `hildale_qty` for in-house. Verify the sync is running on schedule. If the Extensiv columns populate (`extensiv_on_hand_snapshot`) but `current_stock` does not, wire the sync to update `current_stock = extensiv_on_hand_snapshot + hildale_qty`.

**Step 4 — Test.** After fixes: open Raw Materials view → 34 components (per bom_items) should all show non-zero `dailyUsage` for any product that has sold in the last 90 days. Critical/Order badges should populate. No code change considered complete until Sammie confirms the page looks reasonable.

**Files expected to change:**
- `server/storage.ts` — `getSkuSalesVelocity` if it's reading the stale table
- `extensiv-sync.js` — possibly, to also write `current_stock` on finished goods
- No UI changes expected to `client/src/pages/raw-materials.tsx`

**No migrations required.** Schema is already correct.

## B-003 — In-House Shipping Batch Delay Email

**Severity:** Medium. Ops task, not data-broken.

### Root cause

`client/src/pages/in-house-shipping.tsx` already has: checkbox selection, batch "Fulfilled In-House" / "Fulfilled by Pyvott" / "Print Labels" actions. **What's missing is a "Send Delay Notification" batch action** that emails customers whose orders are >3 days old and still unfulfilled.

The `@sendgrid/mail` dependency is already in `package.json`. Sendgrid infrastructure exists.

### Fix plan

**Step 1 — New server route: `POST /api/sales-orders/in-house/send-delay-notifications`.** Accepts `{ orderIds: string[] }`. For each:
- Look up customer email on `sales_orders.customer_email`
- Render a delay notification template (see Step 2)
- Send via existing Sendgrid client
- Log result to `system_logs` with event type `delay_notification_sent`, `delay_notification_failed`, or `delay_notification_skipped_no_email`
- Return `{ sent: n, failed: n, skipped: n, errors: [] }`

Rate-limit the send to 10/sec to respect Sendgrid limits.

**Step 2 — Email template.** Stacy-voice per the brand rules in global memory (no em dashes, max 1 exclamation, "Go win your ground war" line, single CTA). Template variables: `{{customer_name}}`, `{{order_id}}`, `{{days_old}}`, `{{expected_ship_date}}`. Route the template through `server/services/` as a module, not inline — matches the existing structure.

**Step 3 — UI.** In `in-house-shipping.tsx` batch-action bar (around line 454-496), add a new button:
- Visible only when selected orders include any with `age > 3`
- Label: `Send Delay Notice ({count})`
- Confirmation dialog lists each customer, order ID, age — and explicitly says no inventory changes will occur
- On confirm, call the new API route, show toast with sent/failed/skipped counts

**Step 4 — Prevent duplicate sends.** Add `last_delay_notification_at` timestamp column to `sales_orders` (migration). Skip orders where notification was sent in the last 72 hours. Show "Notified 2d ago" badge on the row.

**Files expected to change:**
- `server/routes.ts` — new route
- `server/services/delay-notification.ts` — new service module
- `client/src/pages/in-house-shipping.tsx` — new button + dialog
- `shared/schema.ts` + drizzle migration — `last_delay_notification_at` column

**Migration required:** yes, single column addition.

## B-005 — Amazon SKU Population

**Severity:** Medium. Dual-phase plan.

### Root cause

`items.amazon_sku` column already exists in schema and is `NULL` for all 66 rows.

**Audited the worktree 2026-04-23.** It contains `amazon-client.ts` (full SP-API client with OAuth token refresh and an `updateInventory(sellerSku, qty)` method) and `amazon-inventory-sync-service.ts`, which is a **PUSH** service — it reads `item.amazonSku` and writes our qty to Amazon. It *depends on* a populated `amazon_sku` mapping and does not populate it. The worktree solves a different problem (outbound inventory sync) than B-005 (inbound SKU population). Useful for future Phase 4 work, but not a solution for B-005.

### Fix plan

**Step 1 — Manual entry first.** Add an inline "Amazon SKU" column to the Products page (`client/src/pages/products.tsx`) with the same Pencil-edit affordance as `raw-materials.tsx` uses for `onHand`. Sammie backfills from Seller Central for the 11 Shopify-facing finished goods. PATCH `/api/items/:id` already supports column updates.

This unblocks the downstream PUSH service in the worktree (once merged, it will have SKUs to push to).

**Step 2 — Automated SP-API pull.** Add a new method to `AmazonClient` in the worktree: `getListings()` that hits SP-API `getListingsItem` or `getCatalogItem` for the seller. Build a companion service `server/services/amazon-sku-sync.ts`:
- Daily scheduled job via `scheduler-service.ts`
- Calls `AmazonClient.getListings()`
- For each result, match on UPC or ASIN against `items.upc` / `items.amazon_asin`
- Update `items.amazon_sku` if empty or changed
- Log to `system_logs` + `integration_health`

**Step 3 — Observability.** Add a tile to the Integrations page showing last Amazon SKU sync time and number of items mapped.

**Files expected to change:**
- `client/src/pages/products.tsx` — inline edit column (Step 1, immediate)
- `server/services/amazon-sku-sync.ts` — new service (Step 2, after worktree merges)
- `server/services/amazon-client.ts` (in worktree) — add `getListings` method
- `server/scheduler-service.ts` — register daily job

**No migrations required.** Column already exists.
